Entendiendo las Piezas: Arquitectura Desacoplada vs. MonolíticaUna duda muy común cuando estamos aprendiendo es por qué necesitamos tantas herramientas distintas. ¿No puedo simplemente subir mi HTML, mi base de datos y mi servidor de Node.js/Python a un solo lugar como se hacía antes con cPanel o un hosting compartido tradicional?Sí, se puede. A eso se le llama Arquitectura Monolítica. El problema del monolito es que si tu base de datos se sobrecarga y se cae, se lleva consigo a tu servidor y a tu página web; todo colapsa al mismo tiempo. Además, es difícil de escalar si de repente te haces viral.Por eso, las Startups modernas usan una arquitectura Desacoplada (o basada en microservicios/componentes independientes). Esto significa separar la "cara" de la aplicación del "cerebro" y de la "memoria". Si una parte falla o necesita más recursos, las demás siguen funcionando con normalidad.🍔 La Gran Analogía del RestaurantePara entender por qué GCP y Vercel no se reemplazan, sino que se complementan en perfecta armonía, imagina que tu aplicación web es un restaurante de alta gama:Vercel / Cloudflare (El Garzón y el Salón de Comedor): Son excelentes para entregar cosas rápidas al cliente y presentar el menú de forma hermosa. Se especializan en el Frontend (HTML, CSS, JS, React, Vue). Tienen una red global (CDN - Content Delivery Network) que hace que tu página cargue en milisegundos. Si un usuario entra desde Chile, Vercel le sirve la página desde un servidor en Santiago; si entra desde Japón, se la sirve desde Tokio. Sin embargo, no están hechos para "cocinar" platos complejos ni almacenar inventario masivo.Google Cloud (La Cocina Industrial): Es donde ocurre el trabajo pesado. Aquí vive tu Backend (tu API) y tus algoritmos complejos. Si hay que procesar un pago, analizar una imagen, enviar correos masivos o hacer cálculos matemáticos pesados, se hace aquí, lejos de la vista del cliente. Es un entorno oculto al público, extremadamente robusto, seguro y capaz de escalar (contratar más cocineros mágicamente si llegan muchos pedidos).Supabase (La Bodega de Ingredientes y el Libro de Cuentas): Tu Base de Datos. Guarda la información de los usuarios, las contraseñas encriptadas, los productos y las transacciones de forma estructurada. Es la única fuente de verdad (Single Source of Truth) de tu aplicación.🏆 Tu "Stack" de Producción Elegido (Nivel Pro)Este es el conjunto de herramientas exacto que configuraremos. Todo ha sido seleccionado meticulosamente para mantener un costo de $0 sin sacrificar calidad ni rendimiento:Dominio: NIC Chile (.cl) ➡️ Único pago anual (~$9.950 CLP). Fundamental para dar confianza a usuarios locales, proteger tu marca y mejorar drásticamente tu SEO (posicionamiento en Google) para búsquedas dentro de Chile.Frontend: Vercel ➡️ Gratis. Ofrece una red global de distribución (Edge network), despliegues automáticos con cada git push y generación de certificados de seguridad SSL gratuitos y automáticos.Backend: Google Cloud Run ➡️ Gratis (hasta 2 millones de peticiones/mes). Funciona con contenedores Docker y tiene la magia del "Escalado a Cero". Esto significa que si nadie usa tu app a las 3 AM, el servidor literalmente se apaga y no consume recursos (ni dinero). Cuando alguien entra, despierta en milisegundos.Base de Datos: Supabase ➡️ Gratis (plan generoso de por vida). Está construido sobre PostgreSQL, el motor de base de datos relacional de código abierto más avanzado del mundo. Te ahorra configurar servidores de bases de datos desde cero y viene con autenticación integrada.🗺️ 2. 🔌 Anatomía de una Petición (El viaje de un clic en milisegundos):El Inicio (Usuario): Alguien en su celular escribe tu-proyecto.cl en su navegador Chrome o Safari.El Directorio (DNS): El navegador pregunta a la red global: "¿Quién tiene la dirección IP de tu-proyecto.cl?". NIC Chile mira su libreta de direcciones y responde: "Ese dominio está alojado en los servidores de Vercel".La Interfaz (Vercel): El servidor más cercano de Vercel realiza un "handshake" de seguridad (SSL) y le envía al celular del usuario el paquete inicial: HTML, CSS, JavaScript y las imágenes. La página dibuja su interfaz.La Acción (Frontend a Backend): El usuario hace clic en el botón "Ver mi Perfil". El código JavaScript de tu Frontend ejecutándose en el celular hace una petición silenciosa en segundo plano (un fetch o petición AJAX) hacia la URL secreta de tu Google Cloud Run (https://api-miapp.a.run.app/perfil).El Cerebro (GCP a DB): Cloud Run despierta, recibe la petición, valida que el usuario tenga permisos (usando un token JWT), se conecta a Supabase de forma segura, ejecuta una consulta SQL para buscar el usuario, extrae sus datos, los formatea en un archivo JSON y los envía de vuelta por el océano de internet hasta el Frontend.El Resultado: El Frontend recibe el JSON y actualiza la pantalla. El usuario ve su perfil. Todo este viaje alrededor del mundo ocurre, literalmente, en 200 milisegundos.🛠️ 3. Paso a Paso: El Gran DespliegueEste es el proceso técnico detallado. La clave del éxito aquí es la disciplina: sigue este orden exacto. Si intentas desplegar el Frontend antes de tener la Base de Datos o el Backend listos, tu página web nacerá rota buscando servidores que aún no existen.Paso 1: Preparación Local (El Puente a la Nube)Antes de apagar tu entorno local, debes preparar tu código. Lo que funciona en el ambiente protegido de localhost rara vez sobrevive al entorno hostil del internet público si no está blindado.El Peligro Crítico de las Contraseñas (.env): Nunca, jamás, bajo ninguna circunstancia, escribas contraseñas, URLs de tu base de datos o "API Keys" directamente en tu código fuente (ej. const dbPassword = "mi_clave_secreta"). Existen bots maliciosos que escanean GitHub las 24 horas del día buscando estos descuidos. Si encuentran una clave de Amazon o Google, pueden usar tu cuenta para minar criptomonedas y generarte una deuda de miles de dólares en horas.La solución: Crea un archivo llamado .env en la raíz de tu proyecto. Pon tus secretos ahí.Abre el archivo .gitignore y escribe .env en una nueva línea. Esto le dice a Git: "Cuando suba mi código a internet, deja este archivo en mi computador".El Puente: Sube todo tu proyecto (Front y Back) a un repositorio privado en GitHub.⚠️ ¡ALERTA DE ERROR COMÚN! El villano llamado CORS: Cuando estabas en local, tu Frontend (ej. http://localhost:3000) y Backend (http://localhost:8080) vivían en la misma casa. Eran amigos. En producción, tu Frontend estará en https://miapp.cl y tu Backend en https://googlecloud.com. Por seguridad estricta, los navegadores bloquean cualquier petición de datos entre dominios distintos. Esto se llama CORS (Cross-Origin Resource Sharing).La analogía del guardia: CORS es como el guardia (bouncer) de una discoteca exclusiva. Si un visitante (una petición) viene de un origen desconocido, no lo deja pasar.Solución Obligatoria: En el código de tu Backend (ej. en Express.js, FastAPI o Django), debes configurar la librería de CORS para decirle al guardia: "Oye, si la petición viene de https://tu-dominio.cl, confía en ella y déjala pasar. Si viene de cualquier otro lado, bloquéala".Paso 2: La Base de Datos en la Nube (Supabase)Migrar la base de datos es el primer paso en la nube, porque tu Backend la va a necesitar apenas nazca y empiece a ejecutarse.Crea un proyecto nuevo en Supabase.com. Selecciona la región geográfica de servidores más cercana a tu público objetivo (ej. São Paulo, Brasil es excelente para latencia baja en Chile).Migración de Tablas y Esquemas: Ve a la sección "SQL Editor" de tu nuevo proyecto en Supabase. Ejecuta exactamente los mismos scripts de creación de tablas (CREATE TABLE...) que usaste en tu entorno local.Credenciales Maestras: Ve a Project Settings -> API. Copia la Project URL y la API Key. Si tu backend necesita privilegios de administrador para saltarse reglas de seguridad, busca la clave service_role. Advertencia: La clave service_role jamás debe llegar a Vercel ni a tu Frontend; es un secreto exclusivo que solo tu backend en Google Cloud debe conocer.Seguridad (RLS - Row Level Security): Este es un paso que muchos olvidan. Supabase usa RLS. Si no activas políticas RLS y alguien descubre tu clave pública, podría descargar o borrar toda tu base de datos desde su terminal. Debes crear políticas que dicten, por ejemplo: "Un usuario solo puede hacer SELECT en la tabla de perfiles si su ID coincide con el ID de la fila".Paso 3: El Backend (Google Cloud Run)Queremos que tu servidor viva en la infraestructura global de Google. Para lograr esto, usamos la magia de los contenedores Docker.¿Qué es un Dockerfile? (La caja de envío): Imagina que quieres enviar tu aplicación por correo. No envías los cables y las piezas sueltas; la metes en una caja estandarizada con instrucciones. Un Dockerfile es un documento de texto en la raíz de tu backend que le dice a Google cómo armar esa caja. Un ejemplo básico para Node.js sería:FROM node:18-alpine
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
EXPOSE 8080
CMD ["npm", "start"]
Cuenta en GCP: Entra a Google Cloud Console. Te pedirá tarjeta de crédito para verificar identidad, pero no te cobrarán mientras te mantengas en la capa gratuita.Crear el Servicio en Cloud Run:En el buscador superior de GCP, escribe "Cloud Run" y haz clic en "Crear Servicio".Selecciona la opción de "Desplegar continuamente desde un repositorio" y conecta tu cuenta de GitHub.Selecciona el repositorio y la subcarpeta donde vive tu backend. GCP leerá tu Dockerfile automáticamente y construirá el contenedor por ti.Autenticación (Crucial): Es vital seleccionar "Permitir invocaciones sin autenticar". Si no marcas esto, Cloud Run exigirá permisos de administrador de Google Cloud para cada petición, bloqueando efectivamente tu página web pública.Inyectando Variables de Entorno: Ve a la pestaña de Variables. Aquí es donde pegas las credenciales de Supabase del Paso 2. Cloud Run inyectará estas variables de forma segura dentro de tu contenedor para que reemplace a tu archivo .env local.Haz clic en Desplegar. Después de unos minutos de compilación, GCP te dará una URL en vivo (ej: https://api-miapp-xyz123.a.run.app). Guarda esta URL celosamente.Paso 4: El Frontend (Vercel)Ahora que tu "Cerebro" (Backend) y tu "Memoria" (Base de Datos) están vivos en internet, vamos a darles un "Rostro".Crea una cuenta en Vercel iniciando sesión con tu cuenta de GitHub.Haz clic en el botón negro "Add New Project" y elige el repositorio donde tienes tu código.Vercel es increíblemente inteligente: escaneará tu código y detectará automáticamente si usas React puro, Vite, Next.js, Astro o Angular, y autoconfigurará los comandos de construcción (ej. npm run build) por ti.La Conexión Final (Variables de Entorno en Front): En la sección de "Environment Variables" de Vercel, debes crear una variable (ej. VITE_API_URL, NEXT_PUBLIC_API_URL o REACT_APP_API_URL, dependiendo de tu framework) y pegar la URL de Google Cloud Run que obtuviste en el paso anterior. ¡Así es como el Frontend sabrá a qué puerta tocar en internet cuando necesite información!Haz clic en Deploy. En aproximadamente 60 segundos, Vercel compilará tu aplicación, la distribuirá por su CDN global y te entregará una URL temporal funcional, como tu-proyecto-sigma.vercel.app.🇨🇱 4. El Dominio: Conectando tu Identidad .clYa tienes una app funcional, felicidades. Pero decirle a un inversionista o a un cliente que entre a tu-proyecto-sigma.vercel.app no proyecta confianza. Necesitamos conectar el nombre oficial que compraste en NIC Chile.Para entender la mecánica de este paso, imagina que un Dominio web es como el nombre de una persona, y el Servidor de Nombres (DNS) es la guía telefónica mundial que traduce ese nombre a un número de teléfono (dirección IP del servidor).A. La Compra Oficial en NIC.clVe a nic.cl, el organismo oficial que administra los dominios .cl en nombre de la Universidad de Chile.Usa su buscador para encontrar tu nombre ideal y asegúrate de que esté libre.Regístrate, llena tus datos fiscales y paga la anualidad.Una vez procesado el pago, quédate en el panel de control de tus dominios en NIC.B. Registrar el Dominio en VercelEn el panel de control de tu proyecto en Vercel, navega a la pestaña Settings y luego a Domains.Escribe tu dominio recién comprado (ej: miapp.cl) y haz clic en "Add".Vercel comenzará a verificar y te mostrará un mensaje de error en rojo: "Invalid Configuration". No entres en pánico, esto es completamente esperado porque NIC Chile aún no tiene idea de que te quieres conectar con Vercel.Justo debajo de ese error, Vercel te dará una lista de Nameservers (Servidores de Nombre de Autoridad). Parecen direcciones técnicas, por ejemplo: ns1.vercel-dns.com y ns2.vercel-dns.com. Cópialos en tu portapapeles.C. La Delegación (El "Apretón de Manos" en el DNS)Vuelve a la página de NIC Chile.Haz clic sobre el nombre de tu dominio para entrar a su administración.Busca la sección que dice "Servidores de Nombre (DNS)" o "Delegación".Borra cualquier servidor por defecto que tenga NIC Chile (generalmente hay líneas vacías o prellenadas con servidores chilenos básicos).Pega los Nameservers que copiaste de Vercel, poniendo un servidor distinto en cada línea.Haz clic en el botón para Actualizar / Guardar datos del dominio.⏳ Paciencia y Propagación DNS: Este proceso de "Apretón de manos" no es instantáneo. Piensa que estás actualizando una guía telefónica global; todos los proveedores de internet del mundo (VTR, Movistar, Entel, etc.) deben actualizar sus memorias caché para aprender tu nueva dirección. Este proceso, llamado Propagación DNS, puede tardar desde 15 minutos hasta 24 horas. Puedes usar herramientas web como whatsmydns.net para ver en tiempo real cómo se va actualizando tu dominio por el mundo.Una vez que termine, Vercel es tan eficiente que automáticamente se comunicará con autoridades de certificación (como Let's Encrypt), generará un certificado SSL criptográfico y lo instalará, dándote el "candadito" verde de conexión segura (HTTPS) de forma totalmente gratuita y renovable automáticamente.✅ 5. Checklist Final: Operaciones del "Día 2"Desplegar no es el final, es solo el comienzo. Antes de lanzar el proyecto al público, y para asegurar que dormirás tranquilo, revisa estos puntos críticos de monitoreo y seguridad:[ ] CORS Verificado: ¿Intentaste hacer una petición a tu backend desde un origen distinto y verificaste que te bloquea, pero que al entrar desde https://midominio.cl sí te deja pasar?[ ] Secretos a salvo: Revisa la historia de tu repositorio en GitHub. ¿Estás absolutamente seguro de que el archivo .env o contraseñas quemadas jamás fueron subidas en un commit anterior?[ ] Inyección de Variables: ¿Ingresé las variables de entorno correctamente tanto en el panel de configuración de Vercel como en la revisión de tu contenedor en Cloud Run?[ ] Prueba End-to-End (E2E): Entra a mi-dominio.cl como si fueras un usuario cualquiera. Crea una cuenta nueva, llena un formulario o haz la acción principal de tu app. Luego, entra al panel de Supabase y verifica visualmente que ese dato se creó de forma íntegra en la base de datos real.[ ] Alertas de Presupuesto (Billing) en GCP: Aunque Cloud Run tiene 2 millones de peticiones gratuitas, es una excelente práctica ir a la sección de "Facturación" (Billing) en Google Cloud y crear una Alerta de Presupuesto. Configúrala para que te envíe un email si tu gasto proyectado supera $1 USD. Así evitarás sorpresas a fin de mes si sufres un ataque DDoS o un error en tu código genera un bucle infinito.[ ] Manejo de Errores (Logs de Producción): Si algo falla en producción, tu pantalla no te mostrará el error amigable de VS Code. Acostúmbrate a revisar la pestaña "Logs" tanto en Vercel (para errores de la interfaz o funciones de red) como en Cloud Run (para errores de caída del servidor backend). Los logs son la caja negra de tu aplicación; aprender a leerlos es la diferencia entre un desarrollador junior y uno profesional.¡Misión cumplida! Has logrado un hito tremendo. Tu aplicación ya no es un frágil proyecto local que depende de tu computadora; ahora es un sistema distribuido en la nube, protegido por una base de datos segura, impulsado por un backend auto-escalable en la infraestructura mundial de Google, y presentado al mundo a través de un frontend ultra-rápido distribuido globalmente. Todo con una arquitectura de primer nivel y con un costo mensual de mantenimiento de $0. ¡A celebrar!
